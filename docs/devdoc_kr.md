# 상태(State)와 필드(Field) 설계 — 단계별 템플릿

객체/클래스에 **어떤 필드가 필요한지**를, 중요한 필드를 빠뜨리지 않는 순서로 결정하기 위한
재사용 가이드입니다. 주니어 동료가 처음 보고도 따라올 수 있도록, 그리고 **온라인 검색 없이도**
작성할 수 있도록 필요한 참조 표를 문서 안에 포함했습니다.

---

## 모든 것의 핵심 아이디어 하나

**필드는 머리로 짜내는 게 아니라, 도출(derive)하는 것입니다.**

필드 누락은 상상으로 상태를 "발명"할 때 생깁니다. 모든 필드를 아래 둘 중 하나에 의해
_존재할 수밖에 없게_ 만들면 누락이 사라집니다.

1. **화면에 보이는 것** — 뒤에 데이터가 없으면 렌더링 자체가 안 됩니다.
2. **상호작용이 읽거나 쓰는 것** — 클릭, 키 입력, 타이머, 충돌.

보이는 요소나 상호작용에 대응하는 필드가 없다면 **그게 바로 누락된 필드**입니다.
모든 단계는 이 빈틈을 눈에 띄게 만들기 위해 존재합니다.

---

## DATA와 STATE 먼저 구분하기 (주니어가 가장 많이 하는 실수)

|                     | **DATA** (정적 설정)                  | **STATE** (변하는 런타임 값)         |
| ------------------- | ------------------------------------- | ------------------------------------ |
| 플레이 중 변하나요? | 절대 안 변함                          | 변함                                 |
| 예시                | 코인 점수값, 스프라이트 크기, 맵 배치 | 플레이어 위치, 점수, 코인 획득 여부  |
| 어디에 두나요?      | `DATA.*`, 상수                        | `STATE.*`                            |
| 불변 조건 필요?     | 불필요 (상수니까)                     | **필요 — 모든 필드에 7개 항목 명세** |

판별법: _"게임 시작 시 두 플레이어가 같은 값을 보나요?"_ → DATA.
_"플레이하면서 값이 달라지나요?"_ → STATE.

## 0단계 — 트리아지(Triage, 30초): 얼마나 자세히 설계할까?

| #   | 질문                                                        | 예? |
| --- | ----------------------------------------------------------- | --- |
| 1   | 이 동작이 **한 번에 2개 이상의 상태**를 바꾸나요?           |     |
| 2   | **돈이나 인벤토리**(플레이어가 "소유"하는 것)를 건드리나요? |     |
| 3   | **씬/맵 전환을 넘어서 상태가 유지**되나요?                  |     |
| 4   | **아직 없는 시스템**을 새로 도입하나요?                     |     |

| "예" 개수 | 위험도     | 무엇을 진행하나                                          |
| --------- | ---------- | -------------------------------------------------------- |
| 3–4개     | **HIGH**   | 전체 시퀀스(1~11단계). 코딩 전 ~45–60분.                 |
| 1–2개     | **MEDIUM** | **1, 4, 5, 6, 9, 10단계만**. ~25분.                      |
| 0개       | **LOW**    | 건드릴 상태의 불변 조건만 다시 읽기. 코딩. 테스트. ~5분. |

**LOW에서도 항상 하기:** 2개 이상 상태를 건드리는 동작은 원자성 사고(4단계);
수정할 상태의 불변 조건 다시 읽기(6단계).

## 전체 시퀀스 한눈에 보기

| 단계                           | 산출물            | 잡아내는 것                         |
| ------------------------------ | ----------------- | ----------------------------------- |
| 1 목적                         | 한 문장           | 범위 확산(scope creep)              |
| 2 보이는 요소                  | 표                | UI 뒤 데이터 누락                   |
| 3 상호작용                     | 표                | 트리거 누락                         |
| 4 트랜잭션                     | 표                | 절반만 적용된 변경 / 데이터 손상    |
| 5 상태 도출                    | 명사 목록         | 발명되거나 누락된 필드              |
| 6 **필드 명세(불변 조건 7개)** | 필드별 블록       | 잘못된 타입, null 크래시, 값 어긋남 |
| 7 연산(Operation)              | 연산별 블록/한 줄 | 잘못된 순서, 실패 처리 누락         |
| 8 버그 시나리오                | 목록              | 출시 전 엣지 케이스                 |
| 9 단위 테스트                  | 목록              | 회귀(regression)                    |
| 10 미해결 질문                 | 표                | 우연히 내려진 결정                  |
| 11 잠금 후 코딩                | 상태 전환         | 설계하면서 동시에 코딩하는 일       |

**여러분이 신경 쓰던 항목들이 들어가는 위치:**

- **Nullable(널 허용)** → 6단계, 불변 조건 #3.
- **버그 시나리오** → 8단계, 4단계 실패 + 6단계 도메인 한계에서 도출.
- **단위 테스트** → 9단계, 6단계 불변 조건 + 7단계 실패 + 8단계 버그에서 도출.

## 진행 예제 (일부러 아주 작게 잡음)

> 플레이어가 필드를 돌아다닌다. 코인 1개에 부딪히면 그 코인을 획득하고 점수가 1 오르며,
> 코인은 사라진다.

머릿속에 담길 만큼 작지만, 움직이는 캐릭터·충돌·트랜잭션(점수와 코인을 함께 바꿈)·
숫자·불리언·널 허용 질문까지 모두 포함합니다.

---

## 1단계 — 목적 (2분)

한 문장. "그리고"가 두 번 필요하면 시스템 두 개다 — 쪼개세요.

**템플릿 — 이걸 복사:**

```
"<주체>가 <무엇을 한다> 그래서 <결과가 된다>."
```

**예시:**

> "플레이어가 코인에 부딪혀 그것을 획득하고, 점수가 1 오른다."

문장이 나오면 멈추세요. 다듬지 마세요.

---

## 2단계 — 보이는 요소 (5분)

목업을 훑으며 **보이는 모든 것**을 적습니다.

**템플릿 — 이걸 복사:**

| 요소 | 종류(img/숫자/배열/버튼…) | 변하나/움직이나? | 이미 있나요? |
| ---- | ------------------------- | ---------------- | ------------ |
|      |                           |                  |              |

**예시:**

| 요소            | 종류           | 변하나/움직이나? | 이미 있나요? |
| --------------- | -------------- | ---------------- | ------------ |
| 플레이어 캐릭터 | img/스프라이트 | 키로 움직임      | 아니오       |
| 코인            | img/아이콘     | 획득 시 사라짐   | 아니오       |
| 점수            | 숫자           | 1씩 증가         | 아니오       |

모든 보이는 것이 한 행을 가지면 멈춥니다. _각 행은 5단계에서 필드를 요구합니다._

---

## 3단계 — 상호작용 (5분)

2단계의 상호작용 가능한 요소마다 모든 트리거를 적습니다. 키보드, 클릭,
**타이머 발화**, **씬 진입**까지 — 클릭만이 아닙니다.

**템플릿 — 이걸 복사:**

| 요소 | 트리거 | 함수 | 입력 | 일어나는 일(상태 변화) |
| ---- | ------ | ---- | ---- | ---------------------- |
|      |        |      |      |                        |

**예시:**

| 요소          | 트리거      | 함수              | 입력 | 일어나는 일             |
| ------------- | ----------- | ----------------- | ---- | ----------------------- |
| 플레이어      | 방향키 누름 | `movePlayer(dir)` | dir  | `player.x/y` 변경       |
| 플레이어↔코인 | 박스가 겹침 | `collect()`       | —    | 코인 획득 표시; 점수 +1 |

2단계의 모든 상호작용 요소가 행을 가지면 멈춥니다.

---

## 4단계 — 트랜잭션 (5분) — 건너뛰지 마세요

3단계 각 행에 대해 묻습니다: **2개 이상의 상태를 바꾸나?** 그렇다면 행을 작성하세요.
마지막 두 열이 곧 미래의 버그 목록입니다.

**템플릿 — 이걸 복사:**

| 연산 | 상태 A 변화 | 상태 B 변화 | A 실패 시 | B 실패 시 |
| ---- | ----------- | ----------- | --------- | --------- |
|      |             |             |           |           |

**예시:**

| 연산        | 상태 A 변화             | 상태 B 변화  | A 실패 시             | B 실패 시                           |
| ----------- | ----------------------- | ------------ | --------------------- | ----------------------------------- |
| `collect()` | `coin.collected = true` | `score += 1` | 중단, 점수 추가 안 함 | (clamp; 메모리 내라 실패 거의 없음) |

상태 2개 → 트랜잭션 → 원자성(6단계 #7)과 변경 순서(7단계)가 필요합니다.
2개 이상 상태를 바꾸는 모든 연산에 행이 생기면 멈춥니다.

---

## 5단계 — 상태 도출 (5분): 명사만 뽑기

**발명하지 말고 — 표를 따라가세요:**

- **보이는 요소마다(2단계):** 어떤 데이터가 바뀌어야 이 요소가 다르게 렌더링되나?
- **상호작용마다(3단계):** 그 트리거가 무엇을 읽고 쓰나?

이미 STATE에 있으면 건너뛰고, 없으면 추가합니다. 반복되는 구조는 객체로 묶습니다. 타입은 아직 안 적음.

**템플릿 — 이걸 복사:**

```
DATA.<NAME> = { … }              // 정적 설정 (있다면)
STATE.<name> = { … } / 값
```

**예시:**

```
DATA.COIN_VALUE = 1                       // 정적
STATE.player = { x, y, w, h }
STATE.coin   = { x, y, collected }
STATE.score  = 0
```

**모든** 2단계 요소가 받쳐주는 상태를 갖고 **모든** 3단계 트리거가 읽거나 쓰는 상태를
가지면 멈춥니다. 이 이중 확인이 필드 누락 방지 관문입니다.

---

## 6단계 — 필드 명세: 불변 조건 7개 (상태당 10분) ★ 핵심

새로운 STATE 객체마다 7개 항목을 모두 채웁니다. `??`라고 적어도 됩니다 — 그건
미해결 질문(10단계)이 됩니다.

**템플릿 — 이걸 복사:**

```
### STATE.<name>
1. Type(타입):          number / string / boolean / array / object {…}
2. Domain(도메인):      어떤 값이 합법인가  (아래 표 참고)
3. Nullable(널 허용):   null/undefined/빈 값이 될 수 있나? 언제? 그 의미는?
4. Relationships(관계): 어떤 다른 상태와 일치해야 하나 (합계·미러·파생)?
5. Lifecycle(생명주기): 언제 생성? 언제 리셋? 언제 소멸?
6. Mutators(변경 함수): 어떤 함수만 이 값을 WRITE 하나? (나머지는 읽기 전용)
7. Atomicity(원자성):   같은 연산에서 함께 바뀌어야 하는 다른 상태는?
```

### 도메인 참조 — 합법적인 값의 형태 (검색 불필요)

타입을 고른 다음, 해당하는 제약을 고릅니다. 이게 "검색 없이 개발 문서를 쓴다"의 핵심입니다.

**숫자** — 부호 규칙 + 정수 규칙 + 범위 + (선택) 허용 집합을 조합:

| 제약 종류     | 쓸 수 있는 옵션                                       | 게임 예시                  |
| ------------- | ----------------------------------------------------- | -------------------------- |
| 부호          | `≥ 0`, `> 0`, `≤ 0`, `< 0`, `≠ 0`, 무제한             | 점수 `≥ 0`; 데미지 `> 0`   |
| 정수 여부     | 정수만 · 소수 허용                                    | 개수=정수; 속도=소수       |
| 범위          | `[min, max]` 포함 · 한쪽만 `[0, ∞)` · 개구간 `(0, 1)` | x `∈ [0, fieldW − w]`      |
| 허용 집합     | 특정 값 `{1, 3, 5}` · N의 배수                        | 타일 크기 `∈ {16, 24, 32}` |
| 특수값 안전성 | **유한**해야 함 (`NaN`, `±Infinity` 제외)             | 계산된 모든 값             |

흔한 숫자 *역할*과 보통의 도메인 (그대로 가져다 쓰세요):

| 역할                 | 보통의 도메인                                      |
| -------------------- | -------------------------------------------------- |
| 개수 / 점수 / 목숨   | 정수, `≥ 0`                                        |
| 위치 (x, y)          | 소수, 필드 경계로 clamp                            |
| 배열 인덱스          | 정수, `∈ [0, length − 1]`, 또는 선택 없으면 `null` |
| 비율 / 채움 %        | `∈ [0, 1]` (또는 `[0, 100]`)                       |
| 타이머 / 카운트다운  | `≥ 0`, 0까지 줄다가 발화                           |
| 속도 / 변화량(delta) | 소수, 부호 무관, 유한                              |

**문자열(string):**

| 제약 종류         | 옵션                                      | 예시                                |
| ----------------- | ----------------------------------------- | ----------------------------------- |
| 열거형(고정 집합) | `{…}` 중 하나                             | `status ∈ {"playing","won","lost"}` |
| ID 참조           | 반드시 **`DATA.X`의 키**여야 함           | `itemId ∈ keys(DATA.ITEM_TYPES)`    |
| 자유 텍스트       | 비어 있지 않음 · 길이 `≤ N` · 문자셋/패턴 | 이름: 1–12자, 영숫자                |

**불리언(boolean):** `true` / `false`만. ⚠ 함정: 의미 있는 _세 번째_ 상태(미확인 /
아직 로딩 안 됨)가 있다면 그건 불리언이 아닙니다 — 열거형으로 바꾸세요.

**배열(array):**

| 제약 종류 | 옵션                               | 예시                      |
| --------- | ---------------------------------- | ------------------------- |
| 길이      | 비어 있지 않음 · `≤ N` · 고정 길이 | items `≤ 6`               |
| 원소      | 각 원소가 자기 도메인을 만족       | 모든 item이 유효한 `type` |
| 유일성    | id 중복 없음                       | item마다 유일한 `id`      |
| 구멍 없음 | `null`/`undefined` 빈칸 없음       | 빽빽한 배열               |

**객체 `{…}`:** 모든 필드가 자기 도메인을 만족(재귀), 필수 키가 존재, 그리고
**필드 간 규칙**을 여기 적습니다 (예: `current ≤ goal`, `x`와 `w`가 박스를 화면 안에 유지).

> **한 줄 레시피:** _기본 타입 → 부호/범위 또는 허용 집합 → 정수냐 소수냐 →
> 다른 상태와의 관계._ 예: `score` = "정수, ≥ 0".

### 예시 (숫자·불리언·널 허용을 보여주는 두 필드)

```
### STATE.score
1. Type:          number
2. Domain:        정수, ≥ 0
3. Nullable when: 절대 null 아님; 0으로 시작 (0은 "비어 있음"이지 null 아님)
4. Relationships: 획득한 코인 수와 같음 (개당 1점)
5. Lifecycle:     게임 시작 시 생성; 재시작 시 0으로 리셋
6. Mutators:      addScore()만 — 충돌 코드는 이걸 호출, score를 직접 쓰지 않음
7. Atomicity:     coin.collected = true 설정과 같은 연산에서 증가
```

```
### STATE.coin
1. Type:          object { x: number, y: number, collected: boolean }
2. Domain:        x ∈ [0, fieldW − w]; y ∈ [0, fieldH − h]; collected ∈ {true, false}
3. Nullable when: 객체는 항상 존재; collected === true가 되면 아무것도 렌더하지
                  않을 뿐(보이지 않음 ≠ null). 레벨에 코인이 없을 때만 null.
4. Relationships: collected === true  ⇒  이 코인에 대해 점수가 정확히 한 번 증가했음
5. Lifecycle:     게임 시작 시 생성; 재시작 시 collected를 false로 리셋
6. Mutators:      collect()가 collected 설정; spawnCoin()이 x,y 설정
7. Atomicity:     collected 전환과 점수 증가가 같은 연산에서 일어남
```

모든 새 상태가 7줄을 가지면 멈춥니다 (일부는 `??`여도 됨).

## 7단계 — 연산(Operation) (10분)

트랜잭션은 전체 블록, 비트랜잭션은 한 줄.

**템플릿 — 이걸 복사:**

```
### <opName>(args)   [트랜잭션]
- 사전 조건: <실행되려면 무엇이 참이어야 하나>
- 실패: <조건> → <중단 / 폴백>
- 변경 순서: (1) … (2) …
    근거: <어떤 단계가 실패해도 상태가 일관되게 남는 이유>
- 다시 렌더: <어떤 뷰가 다시 그려지나>

- <simpleOp>(args): <한 줄 — 무엇을 읽고 쓰나, 실패 모드 없음>
```

**예시:**

```
### collect()   [트랜잭션]
- 사전 조건: coin.collected === false  그리고  플레이어 박스가 코인 박스와 겹침
- 실패: 코인이 이미 획득됨 → 중단, 점수 추가 안 함
- 변경 순서: (1) coin.collected = true   (2) addScore(DATA.COIN_VALUE)
    근거: 플래그를 먼저 뒤집으면, 같은 프레임의 재충돌이 사전 조건에서 걸려
    점수가 두 번 더해질 수 없음.
- 다시 렌더: 코인 레이어, 점수

- movePlayer(dir): player.x/y 기록, 필드 경계로 clamp, 실패 모드 없음
```

모든 연산이 블록이나 한 줄을 가지면 멈춥니다.

## 8단계 — 버그 시나리오 (5분): 상상하지 말고 앞 표에서 읽어내기

각각은 앞 단계의 표에서 기계적으로 나옵니다.

**템플릿 — 이걸 복사:**

| 출처 | 버그 시나리오 | 가드(방어) |
| ---- | ------------- | ---------- |
|      |               |            |

**예시:**

| 출처                          | 버그 시나리오                        | 가드(방어)                             |
| ----------------------------- | ------------------------------------ | -------------------------------------- |
| 4단계 "A 실패 시"             | 이중 획득 (충돌이 2프레임 연속 발화) | 먼저 `collected` 플래그 확인           |
| 6단계 #2 도메인(하한)         | 점수가 음수가 됨                     | 여기선 없음, 페널티 생기면 0으로 clamp |
| 6단계 #4 관계                 | 점수는 더해졌는데 코인이 안 사라짐   | 한 연산에서 플래그+점수 함께           |
| 2단계(스프라이트 vs 히트박스) | 충돌 박스 ≠ 보이는 스프라이트 크기   | `w/h` 명시적으로 정의; AABB 테스트     |

모든 `??`, 도메인 한계, 트랜잭션 실패가 시나리오를 가지면 멈춥니다.

## 9단계 — 단위 테스트 (코드 전/함께 작성)

불변 조건 1개당 + 버그 시나리오 1개당 단언(assert) 하나. 이미 다 나열돼 있으니 문장으로만 옮기면 됩니다.

**템플릿 — 이걸 복사:**

```
<호출 / 조건>  →  <기대 상태>          (출처: 불변 조건/단계)
```

**예시:**

```
collect() 겹친 상태에서        → score === 1 그리고 coin.collected === true   (연산 + #7)
collect() 이미 획득된 상태에서 → 무동작(no-op), 점수 그대로                   (8단계 이중 획득)
movePlayer 경계 밖으로         → x가 [0, fieldW − w]로 clamp 됨               (#2 위치)
AABB(겹친 박스) === true; (떨어진 박스) === false                            (8단계 히트박스)
```

테스트를 못 쓰겠다면, 불변 조건(6단계)이 충분히 구체적이지 않은 것 — 거기서 고치세요.
